# Мониторинг нагрузки (ядро продукта)

Документ описывает доменную логику — то, ради чего приложение существует для тренера по
физподготовке. Расчёты живут в `lib/sports/` как чистые функции под юнит-тесты.

## Зачем

Продукт — **аналитика и наглядное планирование нагрузки**, а не просто хранилище GPS-отчётов.
Полевой сбор RPE закрыт отдельной offline-PWA автора; здесь данные анализируются и планируются
на десктопе.

## Внутренняя и внешняя нагрузка

Две независимые оси. **Деградируют по отдельности** — это важно для устойчивости к пропускам:

| Ось | Источник | Что считаем |
|-----|----------|-------------|
| Внутренняя | RPE игрока | sRPE (session-RPE) |
| Внешняя | GPS-датчики | дистанция, high-speed, ускорения, HR |

- Отказ датчиков → внешние метрики `null`, но при наличии RPE **sRPE всё равно считается**.
- Забыт RPE → внутренняя нагрузка неизвестна, но внешние метрики (если есть GPS) остаются.

> **MVP-оговорка:** колонки внешних метрик в `player_trainings` пока `notNull` — `null`-метрики при
> отказе датчиков **не хранятся** в MVP (импорт требует полных метрик). Ось RPE деградирует независимо
> уже сейчас (`rpe`/`sessionLoad` nullable). `null` внешних метрик — post-MVP (миграция). См. DATA_MODEL.

## sRPE (метод Foster)

```
session_load (AU) = RPE (1–10) × duration_minutes
```

- Длительность хранится в **секундах** (`integer`), в минуты переводится на лету.
- Денормализуется в `playerTrainings.sessionLoad`. Любая запись `rpe` (импорт, ручной ввод,
  правка задним числом, позже PWA-импорт) идёт через **единый** `upsertPlayerTraining`
  (`lib/player-training.ts`), который всегда пересчитывает `sessionLoad` — без дрейфа по точкам входа.
- `sessionLoad = null`, если RPE отсутствует (сохраняет семантику пробела, см. ниже).

## Семантика пропусков (критично для корректности)

Три состояния игрока в конкретный день. **Состояние (3) НИКОГДА не схлопывается в 0** — иначе
занижаются все производные метрики (ACWR, monotony):

| # | Состояние | `sessionLoad` |
|---|-----------|---------------|
| 1 | Нет сессии (отдых) | `0` — легитимный ноль |
| 2 | Сессия есть, RPE есть | посчитан |
| 3 | Сессия есть, RPE нет | `null` — пробел, не ноль |

Частные случаи:
- **Отказ датчиков:** внешние метрики `null`, сессия помечается «без внешних метрик»; sRPE живёт.
- **Зал (`gym`):** RPE **опционален**. Внесён → входит в sRPE; не внесён → `null` (пробел),
  автоматически не учитывается в аналитике. Отдельной логики не требует — работает на семантике выше.
- **Индикатор полноты данных** показывается везде, где есть агрегаты («неделя: 5/6 сессий с RPE»),
  чтобы тренер доверял числу.

## Микроцикл MD-

`trainings.category` несёт matchday-относительную периодизацию: `MD-4`, `MD-3`, `MD-2`, `MD-1`,
`GAME`, `MD+1` и т.д. Недельное представление строится вокруг неё — нагрузка по дням относительно
матча. Это каркас «наглядного планирования».

> **Граница дня.** `trainings.date` хранится в UTC; день/неделя считаются по «дню команды» через
> хелпер в `lib/time.ts` (не инлайн). Иначе группировка по MD-/неделе дрейфит на границе суток.

## ACWR / monotony / strain (фаза сразу за MVP)

Модель данных под это готова уже в MVP (секунды, ясные `null`); реализация — отдельной фазой.

- **ACWR** (acute:chronic workload ratio) — на **EWMA** (экспоненциально взвешенное среднее),
  а не на скользящем среднем: терпимо к пробелам, не требует чистого окна 28 дней, взвешивает
  свежие дни. Используется как флаг риска (зелёная зона ~0.8–1.3, риск вне неё).
- **Monotony** = `mean(daily load) / SD(daily load)` за неделю.
- **Strain** = `weekly load × monotony`.
- На пробелах: предлагать добить RPE задним числом либо исключать день из статистики —
  **не подставлять 0**.

> **Отложенный риск (вернуться перед этой фазой):** EWMA работает на посуточном ряду нагрузки
> игрока. Отдых = `0`, но **строки нет** → отсутствие строки неоднозначно (отдых vs не-внесли).
> Шейпинг `lib/sports/series.ts` (дни → ряд с явным днём отдыха) отложен; учесть при реализации ACWR.

## Что НЕ сохраняем

- Агрегатные строки из `.xlsx` (`Team`, строки по позициям) — пересчитываются на лету в UI.
- Легенда зон из `.xlsx` (пороги acc/dec/HR-зон) — используется для виджетов, не хранится в БД.
